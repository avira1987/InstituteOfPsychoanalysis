# گزارش تشخیص لبهٔ شبکه (lms vs آنیستیتو)

تاریخ بررسی: بر اساس تست‌های انجام‌شده از اینترنت و از داخل VM `tpi-ims`.

## جدول مقایسهٔ پاسخ HTTP

| مقصد | از اینترنت (curl) | Server در هدر | نتیجه |
|------|-------------------|-----------------|--------|
| `http://80.191.11.129/` | بله | **openresty** | 404 HTML پیش‌فرض nginx |
| `http://80.191.11.134/` | بله | **Apache/2** | 200 (سایت اصلی) |
| `http://127.0.0.1/` با `Host: lms.psychoanalysis.ir` | فقط **داخل VM** | **Apache/2.4** | 302 به `/anistito/` |
| `http://80.191.11.129/` از **داخل VM** | — | — | **timeout** (به IP عمومی روی پورت 80 از داخل LXC/VM جواب نمی‌آید) |

## نتیجهٔ کلیدی

- روی VM آنیستیتو (`192.168.91.9`) فقط **Apache** روی `:80` است و VirtualHost **`lms.psychoanalysis.ir`** درست است.
- روی **همان آدرس عمومی** که DNS معمولاً برای lms استفاده می‌شود (**`80.191.11.129`**)، از اینترنت پاسخ **`openresty`** می‌آید — یعنی **لایهٔ جدا از Apache داخل پروژه** است.
- IP **`80.191.11.134`** (سایت اصلی) پاسخ **Apache** می‌دهد؛ یعنی **دو IP عمومی، دو سرویس وب متفاوت** هستند.

بنابراین **404 OpenResty نشان‌دهندهٔ باگ داخل کد آنیستیتو نیست**؛ درخواست به وب‌سرور داخل VM نمی‌رسد یا به ماشین دیگری می‌رسد که OpenResty است.

## راه‌حل‌های ممکن (زیرساخت)

1. **تغییر DNS** `lms.psychoanalysis.ir` به **`80.191.11.134`** (همان لبه‌ای که Apache اصلی است) و افزودن **vhost / location** برای `lms` با `proxy_pass` به `http://192.168.91.9:80` (یا مستقیم به `:3000` اگر فایروال اجازه دهد).

2. **ویرایش OpenResty** روی مسیری که الان **`80.191.11.129:80`** را جواب می‌دهد: `server_name lms.psychoanalysis.ir` و `proxy_pass` به IP داخلی VM آنیستیتو.

3. **DNAT واقعی** روی روتر/Proxmox: ترافیک **`80.191.11.129:80` → `192.168.91.9:80`** به‌جای سرویس OpenResty فعلی روی آن IP.

## تست‌های پیشنهادی برای ادمین شبکه

- روی لبه: `curl -v http://127.0.0.1/ -H 'Host: lms.psychoanalysis.ir'` اگر به OpenResty می‌خورد، همان‌جا باید upstream تنظیم شود.
- مشخص کردن اینکه **کدام ماشین** به اینترفیس با IP **`80.191.11.129`** bind است و **چه پروسسی** پورت 80 را باز کرده (`ss -tlnp` روی آن میزبان).
